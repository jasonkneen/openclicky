# OpenClicky Audit ‚Äî 2026-06-11

Scope: full-repo audit of correctness, security (network surface, input handling, secrets), error handling, performance, test coverage, and dead code. ~79k lines of Swift across the `cursor-buddy` app target, plus bundled agent resources, scripts, and docs.

Verification constraints honored: no `xcodebuild` from the terminal (per `CLAUDE.md`); changed files verified with `xcrun swiftc -parse` and `bash -n`.

## Summary

OpenClicky is a macOS menu-bar AI companion (SwiftUI/AppKit) with push-to-talk voice, screen-aware guidance, a local HTTP/SSE external-control bridge on `127.0.0.1:32123`, and an embedded Codex agent runtime. Overall hygiene is good: no committed secrets, API keys migrated to the Keychain, log redaction in place, the bridge binds loopback-only with constant-time token comparison. The standout problem was that the bridge has been **dead by default since token enforcement landed (commit `321b464`, 2026-05-23)**: nothing provisions a token, nothing hands it to agent sessions, and the bundled skills/scripts/docs don't send it ‚Äî so every authenticated bridge call 401s.

## Findings

### High

1. **External-control bridge dead-by-default since token enforcement** ‚Äî FIXED
   - `cursor-buddy/OpenClickyExternalControlBridge.swift:326` ‚Äî `hasValidBridgeToken` fails closed when no token is configured (correct), but:
   - `cursor-buddy/AppBundleConfiguration.swift:146` ‚Äî `externalControlBridgeToken()` only *reads* config; no code path ever generates a token, and there is no Settings UI for it (repo-wide grep for the defaults key finds no view).
   - `cursor-buddy/CodexProcessManager.swift:40-62` ‚Äî agent process environment never received `OPENCLICKY_BRIDGE_TOKEN`.
   - `AppResources/OpenClicky/OpenClickyBundledSkills/openclicky-screen-control/SKILL.md` ‚Äî only the scribble/highlight examples sent the token; all other curl examples (cursor, cursors, caption, screenshot, speak, clear, mcp/*, events) omitted it.
   - `AppResources/OpenClicky/OpenClickyBundledSkills/openclicky-screen-tour/SKILL.md:73` ‚Äî the Python `post()` helper sent no auth header.
   - `scripts/test-external-control-bridge.sh` ‚Äî live bridge tests sent no token, so they fail against any post-May-23 build.
   - Fix: auto-generate a random token on first use and persist it to the Keychain (`AppBundleConfiguration.ensureExternalControlBridgeToken()`), inject it as `OPENCLICKY_BRIDGE_TOKEN` into agent sessions (`CodexProcessManager.start`), and update both bundled skills, the README, and the bridge test script to send `Authorization: Bearer $OPENCLICKY_BRIDGE_TOKEN`. The bridge stays fail-closed for callers without the token.

2. **Bridge `/click` silently force-enables (and persists) native computer use** ‚Äî NOT CHANGED, recommend follow-up
   - `cursor-buddy/CompanionManager.swift:2413-2415` ‚Äî `clickExternalControlPoint` calls `nativeComputerUseController.setEnabled(true)` when the user has it disabled, and `cursor-buddy/OpenClickyComputerUseRuntime.swift:41` persists that to UserDefaults ‚Äî one bridge call permanently flips a user-facing safety setting (default is off).
   - This contradicts `README.md` ("The bridge is intentionally non-invasive"). Recommended: return `403` when native computer use is disabled instead of enabling it. Left unchanged because it alters a documented endpoint's behavior; product owner should decide.

3. **Agent runtime posture: no sandbox, no approvals, keyring password in env** ‚Äî CITED (likely intentional)
   - `cursor-buddy/CodexProcessManager.swift:33-34` ‚Äî Codex runs with `approval_policy="never"` and `sandbox_mode="danger-full-access"`.
   - `cursor-buddy/CodexProcessManager.swift:113-117` ‚Äî `GOG_KEYRING_PASSWORD` is injected into every agent process, so any spawned agent can decrypt local Google Workspace credentials. Worth a doc note at minimum.

### Medium

4. **Data race on pre-provider audio buffer** ‚Äî FIXED
   - `cursor-buddy/BuddyDictationManager.swift:784` appended to `audioBuffersCapturedBeforeProviderReady` from the AVAudioEngine tap thread while `:748-750` (flush) and `:681` (teardown) mutate the same array on the main actor, with no synchronization. Swift `Array` is not thread-safe; concurrent append/removeAll is undefined behavior. Fix: guard all three access sites with an `NSLock` and flush via snapshot-and-clear.

5. **Retain cycle keeps the bridge server alive after `stop()`** ‚Äî FIXED
   - `cursor-buddy/OpenClickyExternalControlBridge.swift:87-96` ‚Äî the stored `listener`'s `stateUpdateHandler` strongly captured `self` (via `self.port`), creating `self ‚Üí listener ‚Üí closure ‚Üí self`. Fix: capture the port value.

6. **SSE connections are never pruned proactively** ‚Äî CITED
   - `cursor-buddy/OpenClickyExternalControlBridge.swift:443-470` ‚Äî `/events` connections are only removed when a later broadcast write fails or the state handler fires; no keepalive frames are sent, so silently-dead connections linger in `sseConnections`. Low impact (loopback-only, token-gated) but a slow `NWConnection` leak for long-running sessions.

7. **`/health` discloses tool inventory and config pre-auth; no Host validation** ‚Äî CITED
   - `cursor-buddy/OpenClickyExternalControlBridge.swift:158-175` ‚Äî unauthenticated `/health` returns the full tool list, capability metadata, and whether the inference proxy is enabled; there is no Host-header check, so a DNS-rebinding page can fingerprint the bridge (though it cannot actuate anything without the token, and simple cross-origin requests cannot carry the auth header). Consider trimming the unauthenticated body to `{ok, name, port}`.

8. **Inference proxy buffers entire upstream responses** ‚Äî CITED
   - `cursor-buddy/OpenClickyExternalControlBridge.swift:365` ‚Äî `proxySession.data(for:)` waits for the complete upstream body, so `"stream": true` requests through `/v1/messages` / `/v1/chat/completions` stall until completion (up to the 600 s resource timeout at `:65`). Disabled by default (`AppBundleConfiguration.swift:133`), so low urgency; implement chunked pass-through if the proxy is promoted.

### Low

9. **Hardcoded personal path baked into the app** ‚Äî CITED
   - `cursor-buddy/CompanionManager.swift:16354` ‚Äî `seedBuiltInShortcutsIfNeeded` seeds `/Users/jkneen/Documents/GitHub/openclicky` into direct-action memory. Guarded by `fileExists` so it is inert on other machines, but it is developer-specific behavior shipping in the product.

10. **Message logs grow without pruning** ‚Äî CITED
    - `cursor-buddy/OpenClickyMessageLogStore.swift:20-26` ‚Äî daily `messages-YYYY-MM-DD.jsonl` files are created but never deleted; disk usage grows unbounded over months. (Redaction at `:290-313` is solid: key-name heuristics plus value patterns for Anthropic/OpenAI/GitHub/Google tokens, JWTs, and bearer strings.)

11. **`TutorTargetClickTracker` global monitor not removed on deinit** ‚Äî CITED
    - `cursor-buddy/CompanionManager.swift:16148-16175` ‚Äî `disarm()` removes the `NSEvent` global monitor, but there is no `deinit` cleanup. Currently safe because the tracker is a `let` on the long-lived `CompanionManager` (`:1522`); a leak hazard if it is ever instantiated per-session.

12. **`CompanionManager` is a 16.6k-line god object** ‚Äî CITED
    - `cursor-buddy/CompanionManager.swift` mixes voice, TTS, overlay, bridge handling, computer use, agents, click tracking, and persistence. Several embedded types (e.g. `TutorTargetClickTracker:16142`, `OpenClickyDirectActionMemoryStore:16220`) belong in their own files. Refactor opportunity, not a defect.

### Positive observations

- No secrets committed; the only `sk-` hits are redaction regexes (`cursor-buddy/OpenClickyMessageLogStore.swift:302`, `cursor-buddy/CodexHomeManager.swift:700`).
- API keys and the bridge token are Keychain-backed with UserDefaults migration (`cursor-buddy/AppBundleConfiguration.swift:274-297,344-352`).
- Bridge binds `127.0.0.1` only (`cursor-buddy/OpenClickyExternalControlBridge.swift:78-81`), caps request bodies at 10 MB (`:143`), clamps durations/travel (`:912-922`), and compares tokens in constant time (`:337-348`).
- Pre-provider audio buffering is bounded at 360 buffers (`cursor-buddy/BuddyDictationManager.swift:287`).
- Console logging avoids raw transcripts/PII (lengths and provider names only, e.g. `cursor-buddy/BuddyDictationManager.swift:751`).
- Analytics is an intentional no-op shim ‚Äî no telemetry leaves the machine (`cursor-buddy/ClickyAnalytics.swift:5-7`).

### Test coverage

- 1,235 lines of tests across 6 files (`cursor-buddyTests/`) versus ~79k lines of app code; coverage focuses on parity models, computer-use parsing, widget state, and visual-guidance overlays. The bridge's pure parsing layer is exposed for tests (`OpenClickyExternalControlBridge.swift:973-987`) but the HTTP/auth path has no automated tests. Tests run only via Xcode (terminal `xcodebuild` is prohibited by repo policy).

## Changes made in this branch

1. Auto-provision the bridge token (Keychain) and inject `OPENCLICKY_BRIDGE_TOKEN` into agent sessions ‚Äî `AppBundleConfiguration.swift`, `CodexProcessManager.swift`, `CompanionManager.swift`.

3. Guard the pre-provider dictation audio buffer with a lock ‚Äî `BuddyDictationManager.swift`.
4. Break the bridge listener retain cycle ‚Äî `OpenClickyExternalControlBridge.swift`.
